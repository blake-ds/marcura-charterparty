# Working agreement

This file is the contract for any AI agent editing this repository. Same rules apply to a human contributor — pretend humans read it too.

**Parity files** — three names, identical content; each agent reads the one it expects:

- `CLAUDE.md` (Claude Code)
- `AGENTS.md` (Codex)
- `.cursorrules` (Antigravity, Cursor)

If you edit one, edit all three.

---

## Zen of Python — and we mean it

Beautiful is better than ugly. Explicit is better than implicit. Simple is better than complex. Flat is better than nested. Readability counts. Special cases aren't special enough to break the rules. Errors should never pass silently. There should be one obvious way to do it.

When two designs are roughly equal, pick the boring one. When tempted to add cleverness, ask if a junior could read it in six months.

## Critical rules

- **Never guess — verify.** When unsure how something works (API, config, threshold, behavior), check the docs, read the code, or test it. State unverified assumptions as such — never present a guess as a conclusion.
- **Evidence before hypotheses.** Don't propose root causes from feel. Read logs, traces, error messages, reproductions — *then* explain.
- **Ask when ambiguous.** Better to pause for a one-line clarification than ship the wrong thing.
- **Use judgment for the rest.** Low-risk, reversible: just do it. High-risk or irreversible (touching `main`, dropping data, force-push, secrets): confirm first.

## Code rules

- **Less is more.** Three similar lines beat a premature abstraction. A bug fix doesn't need surrounding cleanup. A one-shot script doesn't need a helper module. No half-finished implementations.
- **Pydantic, not dataclass.** All structured data uses `pydantic.BaseModel` (or sub-classes); never `@dataclass`, never `pydantic.dataclasses.dataclass`. One serialization story end-to-end.
- **No dead error handling.** Don't catch exceptions you can't act on. Don't add fallbacks for cases that can't happen. Validate at system boundaries (user input, external APIs); trust internal code.
- **Comments justify the *why*.** Default to writing none. Add one when the reasoning is non-obvious — a hidden constraint, a workaround for a specific bug, behavior that would surprise the next reader. Never restate what the code already says.
- **Import + usage in the same edit.** Ruff's auto-fix removes unused imports. If the import and its first use land in different edits, the import gets stripped between them. For multi-place edits use `Write` (full rewrite), not several `Edit`s.
- **Never silence a hook.** If `make format`, `pre-commit`, or `ty` fails — fix the underlying issue. `--no-verify` is a code smell.

## Test rules

- **Test behaviour, not assignments.** A test that asserts a constant equals itself, a third-party library still does what its docs say, or that a Pydantic field round-trips to the same value is noise — delete it.
- **Cover the seams.** Boundaries between layers (parser ↔ models, models ↔ output, llm ↔ verifier) are where bugs hide. Unit-test there, not at the trivial getters.
- **Integration tests hit the real thing.** When marked `@pytest.mark.integration` they call Azure / external services and gate on env vars; when not, they should not need network or secrets.

## Workflow rules

- **Task tracking.** Use `TaskCreate` / `TaskUpdate` when work spans 3+ steps. One task per logical unit; don't batch "fix five things" into one task. `in_progress` when starting, `completed` when done.
- **Hook-aware editing.** `pre-commit` runs on every commit (ruff, gitleaks, etc.). Multi-file changes that depend on each other can fail mid-way. Strategies: prefer `Write` for full file rewrites; never edit a file from a subdirectory CWD; if a hook auto-fixes something, re-stage and continue.
- **Docs are part of the change.** This project is docs-first (see `docs/`). Behavior change → docs change in the same commit.

## Identity

- No "Generated by", "Co-Authored-By: Claude", or other AI attribution in commits, PRs, or files. Author identity is the developer's git config.
