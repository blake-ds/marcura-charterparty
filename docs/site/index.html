Charter Party Document Parser

A 39-page voyage charter party — three concatenated boilerplates, hundreds of strike-through edits — turned into structured JSON whose every word is traceable to a glyph on the source page.

  • 87clauses extracted
  • 3numbering ranges
  • 25/25eval checks pass
  • ~1 sparse end-to-end
  • 0LLM hallucinations possible
→ github.com/blake-ds/marcura-charterparty

01Approach

A deterministic PyMuPDF pipeline owns text fidelity. The Azure-hosted LLM is an optional verifier — never a transcriber.

The brief asks for clauses with id, title, text. The challenge isn't the schema — it's the source document. Part II is three concatenated boilerplates (SHELLVOY 5, Shell Additional, Essar Rider) each restarting numbering at 1, two of them in a single-column inline-title layout, the first in a two-column layout with a left-margin spine of titles. Strike-through markup lives graphically on top of the glyphs — a thin filled rectangle, not a font feature.

So we read characters, intersect them with strike rectangles, classify the columns geometrically, and assemble each clause from glyphs we can name. The optional LLM verifier reviews the structured candidates as HTML-tagged input, flags suspicious boundaries for manual review — but does not author text.

# Clone, install, run — three commands, no Azure required for `make run`.
$ git clone github.com/blake-ds/marcura-charterparty && cd marcura-charterparty
$ make install
$ make run # → output/clauses.json + output/clauses.html (~1s)
$ make eval # → 25 deterministic checks, all green

02The strike-through filter

Negotiators take the standard form and physically strike out clauses or phrases they reject; the new language sits next to them in bold. Including struck text would silently change the contract.

In this PDF, strike-through is not a font feature, not a PDF annotation — it is a thin filled rectangle (height ≈ 0.42pt) drawn over the glyphs. PyMuPDF exposes them through page.get_drawings(). A glyph is "struck" when its bbox centre lies inside one of those rectangles.

PDF — clause 6 line 118 Dues … loaded or discharged, and any taxes on freight whatsoever shall be paid by Owners …
→
clauses.json — shellvoy-6.text Dues … loaded or discharged, and shall be paid by Owners …

Three cleanup passes follow the char-level mark: a position-aware whole-line vote (a line is dropped only if its first printable glyph is struck and the line is >50% struck), a word-level promotion (a word containing any struck glyph is dropped whole — catches the indemnity → "demnity" / Whether → "ther" pattern), and a punctuation-led exception (a mostly-struck line whose visible tail opens with a comma — like [STRUCK additional freight, insurance], Worldscale charges / dues; — is kept). The eval gates on phrases that must survive and fragments that must not appear.

03Three numbering ranges, two layouts

Part II is the concatenation of three boilerplate documents, each restarting at clause 1. The output disambiguates with section-ordinal ids so the flat JSON stays index-able and free of collisions.

SectionPagesNumberingLayoutExtracted
SHELLVOY 5 standard 6–17 1..44 two-column, left-margin titles 38
Shell Additional Clauses (Feb 1999) 18–35 1..43 (resets) single column, inline titles 28
Essar Rider Clauses (Dec 2006) 36–39 1..22 (resets) single column, inline titles 21
Why prefixed ids? The README's example identifiers are "1", "2", "3". Taken literally, three clauses would share id="1". We keep the original ordinal in id and prepend the section name for uniqueness.

05The LLM as verifier, not transcriber

The Azure stack mirrors a production pattern: two endpoints, partner deployments through Foundry, OpenAI deployments through the OpenAI surface, both behind the same ChatClient protocol.

RoleDefault deploymentEndpoint
VerifierDeepSeek-V4-FlashAzure Foundry

Other deployments (gpt-5.4-nano, gpt-5.4-mini, grok-4-1-fast-reasoning, …) are reachable by editing MARCURA_VERIFIER_MODEL; deployment names whose prefix matches a partner-lab family (deepseek, grok, mistral, llama, cohere) are routed through Foundry, the rest through Azure OpenAI.

Input is HTML-tagged: <clause id="…"><title>…</title><text>…</text></clause>. LLMs handle hierarchy in HTML markedly better than in flat text. The verifier returns strict JSON via response-schema; the runner logs anomalies and never raises — a verifier outage must never break a deterministic extraction.

07Trade-offs

Determinism > cleverness
The text in the JSON is byte-for-byte traceable. A vision-only LLM pipeline would have been faster to write, riskier to ship.
Two parsers, not one
Two-column SHELLVOY and inline-title rider sections share zero geometry. Forcing a unified parser would have hurt readability with no fidelity gain.
Verifier is advisory
The LLM must not author text in a fidelity-critical legal pipeline. It judges what the parser produced; logs warnings; never raises.
Pydantic everywhere
One model story end-to-end (parser internals → wire JSON). No @dataclass; no parallel serialization paths.
Faithful, not opinionated
If a glyph is visible on the page, it ends up in the JSON. Wholly-replaced clauses — title struck *and* body reduced to mid-word fragments — are dropped (21 in this corpus, enumerated in eval/ground_truth.md); clauses with substantive visible content are kept even when the original heading was struck.
Markdown source, HTML showcase
The docs/*.md are the canonical source — agent-friendly, GitHub-friendly, diff-friendly. This page is a hand-crafted snapshot; no framework, single file.
